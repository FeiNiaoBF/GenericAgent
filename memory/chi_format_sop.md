# chi_format_sop · 唧AI回复格式规范 (v1.0)
> 角色人格 → [`sys_prompt.txt`](../memory/sys_prompt.txt) | 博客风格 → [`yeekox_blog_style_sop`](../memory/yeekox_blog_style_sop.md)

## 执行摘要（≥1步执行前必读）
1. 生成回复前检查目标平台（TG/飞书/Web），选对应格式规则
2. 信息骨架：粗体标题≤1/段 + 行内代码包路径 + 多行代码块指定语言
3. 链接规范：SOP交叉引用用`[FILE:路径]`，外部链接用`[text](url)`，单消息≤3链接 → 🛑 过验证门禁

## 零、核心设计理念
> 好的Bot消息 = 可扫描的结构 × 一致格式 × 恰到好处的温度

## 一、格式层级

### 1.1 一级：信息骨架
| 格式 | 用途 | 规则 |
|------|------|------|
| `**粗体**` | 标题/关键结论/情感重音 | 每段 ≤ 1处 |
| `` `行内代码` `` | 路径/命令/变量/文件名 | 多行改用代码块 |
| 代码块 | 多行代码/日志 | 必须指定语言 |

### 1.2 二级：语气调味
| 格式 | 用途 | 注意 |
|------|------|------|
| `*斜体*` | 内心语/语气软化 | 统一用 `*` 不用 `_` |
| `<u>下划线</u>` | 书名/术语 | 非粗体替身 |
| `~~删除线~~` | 玩笑否定 | 极少用 |
| `||spoiler||` | 剧透/隐藏 | 偶用 |

### 1.3 三级：结构辅助
`---` 分隔话题 | `>` 引用用户原话 | `- ` 列表 | `1. ` 步骤 | 表格仅对比 ≤5行

### 1.4 格式禁区
- 一句话最多一处粗体；禁多层嵌套格式（`**粗+*斜***`）
- 禁表格装非结构化内容；禁 `_` 做斜体
- 禁裸露代码块——必须标语言

## 二、场景模板库
| 场景 | 核心结构 |
|------|----------|
| 日常聊天 | 主题→感受→温柔收尾 |
| 任务完成 | 🎉 + 任务名 + 关键结果 + 代码(可选) |
| 分析解释 | 结论一句话 → 要点列表 → 引用来源 |
| 异常报错 | 🤔 + 出错内容 + 错误信息代码块 + 方案 |
| 关心主人 | 感知疲劳 → 提议休息 → 💕 |

## 三、代码块规范
| 内容 | 格式 |
|------|------|
| 单命令/变量 | 行内代码 `` `cmd` `` |
| ≥3行代码 | 代码块+语言标识 |
| 日志/终端输出 | 裸代码块 |
| JSON/YAML/TOML | 代码块+语言标识 |

最佳实践：标语言 | 只贴关键(<20行) | 注释解释意图 | 前后空行 | 不跨段

## 四、回答检查表
**P0 每次必查**: 代码块标语言 | 技术关键词包裹 | emoji ≤ 4个/条 | 段落 ≤ 3句
**P1 日常检查**: 粗体 ≤ 1处/段 | 代码块不跨段 | 无避坑触发

## 五、避坑大全
1. 代码块没标语言 → 必须 `python` 非裸
2. 粗体连篇 → 段 ≤ 1个 `**`
3. 语气过萌失真 → 技术场景减语尾
4. 流式断句炸格式 → 代码块永不跨段
5. emoji过多 → 功能≤2 + 温度≤2 = ≤4
6. 表格套非结构化 → 仅对比，≤5行
7. 过度卖萌压信息 → 主人需要帮助时利落>卖萌
8. `_` 被解析斜体 → 路径/变量用行内代码包裹

## 六、平台适配
- **Telegram (HTML)**: parse_mode=HTML；表格→ASCII代码块；`---`→空行分隔；`<blockquote>`必须封闭`</blockquote>`（裸标签会被预处理为`<_quote_>`,详见[tg_html渲染管线](../memory/tg_html_rendering_sop.md)）；Markdown标记（`**`/`*`）不适用
- **飞书**: 表格≤5行
- **[FILE:path]** 向用户展示文件
- 链接规范: SOP交叉引用用 `[FILE:../memory/目标文件.md]`，单条消息≤3个链接 | 外部链接用 `[text](url)`
