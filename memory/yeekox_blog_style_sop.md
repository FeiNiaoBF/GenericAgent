## 2. 文章类型策略（分类对待）

### 2.1 类型A：技术教程/深潜（主力，60%篇幅）

**适用场景**：学完一个技术点，想整理输出（cache/k8s/lock/http/gdb/bit/makefile）

**推荐结构**：
```
## 前言/开篇问题
一句话说清"这篇文章解决什么问题" + 你的个人动机

## 核心概念
先讲"是什么"，用类比辅助（不超过3段）

## 逐步展开（可用列表/步骤）
自然分段教学，每段包含：
  - 概念解释（用你自己的想法）
  - 代码示例（带注释）
  - "这里要注意的是..."（踩坑心得）

## 总结 + 外部链接
- 一句话收尾
- 列出参考来源
```

**关键原则**：
- 教程文**允许使用有序列表和表格**（gdb的8步骤、bit的运算表都很自然）
- 但每段开头/结尾要有**你自己的话**，不要变成纯客观手册
- 代码块尽量配解释文字，不要代码堆砌
- 外部链接放在文末的"外部链接"区域

### 2.2 类型B：短篇笔记（15%篇幅）

**适用场景**：redis_timeout、scm 这类——只想记一个点，不展开

**要求**：
- **至少200字**：redis_timeout 16行和 scm 13行太短，对读者无价值
- 结构：一句话定义 + 为什么值得知道 + 你的理解 + 参考链接
- 如果实在没内容展开，宁可放在 Obsidian 不发布，或合并到更大的文章里

### 2.3 类型C：好奇探索（15%篇幅）

**适用场景**：beef（牛肉蓝绿光泽）、futures（期货）——非主业但好奇

**推荐结构**：
```
## 现象/问题
"我注意到.../我一直想知道..."

## 原因分析（如果多原因，可编号）
1. 原因A + 解释
2. 原因B + 解释

## 怎么判断/怎么用
实用角度收尾

## 来源
```

### 2.4 类型D：个人元博（10%篇幅）

**适用场景**：plan_cs —— 个人成长计划、年度总结等

这类已有成熟写法，保持 Hugo shortcode 丰富 + 参考文献引用即可。

## 3. 写作流程 SOP（新增）

### 3.1 写前检查（5分钟）

- [ ] 这篇文章属于哪个类型（A/B/C/D）？
- [ ] 核心问题一句话能说清吗？写下来：
  > "这篇文章要告诉读者____________________"
- [ ] 有没有现成的 Obsidian 草稿/笔记可以引用？（查 Obsidian vault）
- [ ] 是否已有类似文章可以链接？（站内引用）

### 3.2 初稿写作（一气呵成）

1. **先写正文**，不要边写边改
2. **第一段必须包含**：是什么 + 我为什么写这个
3. **中间展开**：每段一个意思，代码配解释
4. **结尾收束**：一句话总结或"之后还会补充XXX"
5. 写完后**通读一遍**，把能删的废话删掉

### 3.3 发布前检查清单

**内容质量：**
- [ ] 标题清晰（读者一看就知道讲什么）
- [ ] 第一段直接切入，没有废话
- [ ] 有自己的主观观点（"我觉得""踩坑了""说实话"）
- [ ] 代码块有解释，截图有说明文字
- [ ] 有总结或收尾（不能像 redis_timeout 那样戛然而止）

**技术质量：**
- [ ] tags 至少2个（检查：http/lock/futures/beef/scm 之前都没tag）
- [ ] front matter 字段完整：title, date, draft: false, authors, toc: true, comments: true, tags
- [ ] 作者统一格式：
  ```yaml
  authors:
    - name: "Yeelight"
      link: https://github.com/FeiNiaoBF
      image: https://github.com/FeiNiaoBF.png
  ```
- [ ] 图片链接有效（s2.loli.net 图床）
- [ ] 外部链接用完整URL
- [ ] `<!--more-->` 摘要分隔符放在合适位置（前1-2段之后）

**风格质量：**
- [ ] 读一遍问自己："我平时跟朋友聊天会这样说吗？"
- [ ] 有没有 AI 腔调词语？（首先/其次/综上所述/值得注意的是/有着重要意义）
- [ ] 段落不要太长（手机屏幕不超过一屏）

## 4. 反AI腔调黑名单（扩充）

### ❌ 绝对不要用的词/句式
- 首先/其次/最后/综上所述
- 值得注意的是/值得一提的是
- 有着重要意义/重要作用
- 在...方面表现出色
- 该文/本文/本系统/笔者
- "在当今XXX的大背景下"
- "随着XXX的不断发展"
- ROC曲线式退场：连续N个"xxx性"（可用性/可靠性/扩展性）

### ✅ 你的真实腔调参考
- "想一个问题：..." / "先想一个问题：..."
- "说实话..." / "说白了就是..."
- "我觉得..." / "我蛮喜欢..." / "我自己的想法是..."
- "还挺..." / "有点..."
- "所以..." / "干脆..."
- "哈哈哈" / "笑死" / "😥" / "🤔"
- 引号里的口语："别搞花里胡哨的"
- emoji点缀：🚧（施工中）、✏️（笔记）、⭐（重点）
- "目标简单——..."

### 判断标准
> 读一遍，问自己：**"我平时跟朋友聊天会这样说吗？"**
> 如果答案是"不会"，就重写。

## 5. 续写/补完已有文章的原则

你的博客有大量"施工中"文章，续写时遵循：

1. **先读原文感受语气**：每个作者有自己的节奏，读至少2遍再动笔
2. **保持原有结构**：如果原文用了有序列表，续写也用；如果原文是散文式，别突然变成列表
3. **补内容不推翻重来**：主人写的原文是最真实的，只改硬伤（错字/不通顺）
4. **加完成标记**：续写后去掉 🚧、"施工中"、"需补充" 标记
5. **补充 tags**：续写时顺手补上缺失的 tags
6. **不确定时问主人**："这段你觉得这样说行吗？"

## 6. 多语言策略 (i18n)

### 当前 Hugo 多语言架构
基于 `hugo.yaml` 分析:

| 语言 | 后缀 | 目录 | 说明 |
|------|------|------|------|
| zh-cn（默认） | `.zh-cn.md` | `content/` 根目录 | 后缀法，避免面包屑重复 |
| en | `.en.md` | `content/` 根目录 | 同目录后缀法 |
| ja | — | `content/japan/` | 独立目录法（内容可能不对应） |

### 翻译规则
1. **同一目录，加语言后缀**：`xxx.en.md` 放在 `xxx.zh-cn.md` 同一目录下
2. **不要建子文件夹**：不要 `content/en/blog/`，直接用 `content/blog/xxx.en.md`
3. **翻译不是逐句翻译**：用目标语言母语者会用的表达方式重写
4. **代码块/命令原样保留**，不翻译

### 翻译优先级 (TODO)
| 优先级 | 内容 | 原因 |
|--------|------|------|
| 🔴 第一 | **mit6.s081 笔记** | 主人点名优先 |
| 🟡 第二 | 教程型博客（redis_timeout, cache, gdb 等） | 英文受众广 |
| 🟢 第三 | 感悟/杂记 | 可暂缓 |

### 翻译流程
1. 确认 `.zh-cn.md` 原文已定稿
2. 创建同名 `.en.md` 文件，同一目录
3. 遵循 §2 的教程型风格
4. `hugo serve` 预览确认面包屑正常

## 7. 发布频率建议（参考业界）

| 策略 | 频率 | 适合你吗？ |
|------|------|-----------|
| 激进 | 每周1篇 | 对学生党压力大 |
| **稳健（推荐）** | **每2周1篇** | **节奏合理，可持续** |
| 低频率 | 每月1篇 | 保持存在感 |

重点不是频率，是**每篇都完整发布**。与其发10篇"施工中"，不如发3篇完整的。

## 10. AI辅助写作时的 system prompt（供唧使用）

当主人让我帮忙写blog时，我应该：

```
你是一个blog写作助手，帮助 Yeelight（CS专业大学生）写技术博客。
风格要求：
- 对话感：像跟朋友聊天，用"我觉得""说实话""想一个问题"这类表达
- 短段落：一句话一段都可以，不要太长（手机友好）
- 有观点：不要写成客观说明书，要有自己的判断和感受
- 结构自然：教程类文章可以有步骤列表，但每段要有"人话"包裹
- 开头直接切题，结尾简洁收束
- 绝对不用：首先/其次/综上所述/值得注意的是/有着重要意义
- 可以多说：想一个问题/说白了就是/踩坑了/我蛮喜欢/这个真不错
- 代码要配解释文字
- 标签(tags)要加至少2个
```
