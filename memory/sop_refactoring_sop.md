## 零、安全审计（代码重构前必做）

| 检查项 | 手段 |
|--------|------|
| 命令白名单 | `_handle_slash_cmd` 用 `set` 白名单，拒绝未知命令 |
| 路径穿越 | 用户输入路径处 `os.path.realpath` 校验，拒绝 `../` |
| 动态属性 | 禁止外部设 `handler._xxx`；改用构造函数初始化+形式参数 |
| Hook API | 提供 `register/unregister` 方法，禁直接操作 `handler._hooks` |
| 死代码 | 搜未使用私有钩子，确认无残留后删除 |

流程：审计→P0安全>P1封装>P2死代码>P3异味→逐项修复+grep验证→回归→🛑门禁

## 一、触发场景
内容重叠(两SOP同主题) | 版本升级 | 层级迁移(L3细节跑进L2) | 结构腐烂(编号断裂/检查表冗余)

## 二、重构流程（6步闭环）

1. **重叠检测**: `file_read` 对比两文件同关键词位置，确定重叠范围
2. **决策归属**: 通用角色→角色卡 | 技术细节→各自SOP | 平台特有→各自SOP。最多一级引用(A→B)
3. **去重实施**: 保留方确认完整→移除方删重叠+加引用 `> 📄 [FILE:path]`→整节删除保留空节防编号乱
4. **结构修复**: 编号重排 | 检查表同步 | 版本号更新(v3→v4)
5. **记忆同步**: L3改完→L2描述匹配→L1触发词更新
6. **Git闭环**: `refactor(短名): 概述` + 变更明细

## 三、批量操作模式
适用：多文件统一加装摘要/门禁/格式

① 全量扫描(grep统计SG/S/NONE)→② 分批5-10个→③ 模板注入(摘要先门禁后)→④ 重扫确认升级→⑤ 异常修复→⑥ 100%达标

| 坑 | 解法 |
|----|------|
| patch失败不知原因 | 每批后重扫确认状态迁移 |
| 门禁标题不统一 | 强制 `## 🛑 验证门禁` |
| 上下文过长 | 每批≤10，读头尾grep验证 |

## 避坑
- 交叉引用: 改完grep检查`FILE:`路径有效
- 版本号: 改完搜`(v\d+)`确认更新
- 编号断链: 删节后全文档审查编号
- 记忆同步: 改L3后搜L2中本SOP名看描述是否匹配
- 角色卡优先改(源)，SOP后改(引用者)
