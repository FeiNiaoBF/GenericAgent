## 零、安全审计优先模式（v3.0 新增）

> **来源经验**：2026-05-08 GA自重构 P0-P3，66+轮。发现安全漏洞优先级远高于功能优化。

**原则**：代码重构前先完成安全审计，再实施功能改动。

### 零.1 审计检查清单

| 检查项 | 手段 |
|--------|------|
| 会话命令白名单 | 所有 `_handle_slash_cmd` 分支必须用 `set` 白名单，拒绝未知命令 |
| 路径穿越防护 | 接收用户输入路径处必须 `os.path.realpath` 校验，拒绝 `../` 逃逸 |
| 动态属性（猴子补丁） | 禁止外部直接设置 `handler._xxx` 私有属性；改用构造函数初始化+形式参数传递 |
| Hook 公共 API | 钩子注册/注销提供 `register_xxx()`/`unregister_xxx()` 方法，禁止外部直接操作 `handler._hooks` |
| 死代码清理 | 搜未使用私有钩子（如 `_done_hooks`），确认无残留后删除 |

### 零.2 安全重构流程

```
① 安全审计 → 搜白名单/路径穿越/动态属性/外部私有属性赋值
② 优先级排序 → P0(安全漏洞) > P1(封装破坏) > P2(死代码) > P3(代码异味)
③ 逐项修复 → 每项修复后验证（grep确认无残留）
④ 回归验证 → 搜所有 frontend 确认无直接私有属性访问
⑤ 🛑 过安全门禁
```

## 一、触发场景

| 场景 | 信号 |
|------|------|
| **内容重叠** | 两份SOP对同一主题有不同描述 → 维护double成本 |
| **版本升级** | SOP功能有重大变更 → 需标记版本号 |
| **层级迁移** | 信息在错误层级（如L3细节跑进L2、L2事实混入L1） |
| **结构腐烂** | 多次patch导致章节编号断裂、检查表冗余 |

## 二、SOP重构流程（6步闭环）

### Step 1：重叠检测 & 范围界定
- 对比相关SOP的**章节标题**和**关键概念**（如「人格核心」同时出现在tg_sop和角色卡）
- 确定重叠范围：整节 / 单一段落 / 一句话
- **工具**：`file_read` 对比两文件同一关键词的出现位置

### Step 2：决策「谁保留，谁引用」
```
有重叠 → 谁更适合做「唯一真相源」？
├─ 通用角色/能力 → chi_character_card（角色卡）
├─ 技术实现细节 → 各自SOP保留
└─ 平台特有规则 → 各自SOP保留
```
**原则**：避免交叉引用链过长（A→B→C），尽量保持一级引用（A→B，B独立）

### Step 3：去重实施
1. 在**保留方**确认内容完整（必要时补充）
2. 在**移除方**：
   - 删除重叠内容
   - 添加引用横幅或文字：`> **📄 [FILE:path]** — 参见[文件名]`
   - 若为整节删除，保留空节并写「已迁移至xxx」避免编号混乱

### Step 4：结构性修复
| 修复项 | 操作 |
|--------|------|
| 章节编号 | `## 二、` → `## 一、` 重排（本文唯一需要全局搜索替换的步骤） |
| 检查表 | 移除已删除内容对应的检查项，或改为引用 |
| 目录/导航 | 如有文内目录，同步更新 |
| 版本号 | v3→v4 形式更新标题（如 `(v4 纯格式版)`） |

### Step 5：多级记忆同步
```
L3 SOP（刚改完）
  ↓ 更新 L2 描述
L2: global_mem.txt → 对应SOP说明行
  ↓ 必要时更新 L1（若功能定位变化）
L1: global_mem_insight.txt → 括号内触发词更新
```

**同步检查表**：
- [ ] L3: SOP修改完成，内容自洽
- [ ] L2: `global_mem.txt` 中对应section的说明是否匹配新版本？
- [ ] L1: `global_mem_insight.txt` 中该SOP的触发词/描述是否需更新？
- [ ] 如有字符卡/角色卡引用：检查 banner/横幅 是否过期

### Step 6：Git 闭环
```
git commit -m "类型(sop名): 行动概述
- 删除xxx(理由)
- 重编号/重构检查表
- 更新L2/L1同步"
```
**类型**：`refactor`(重构) / `feat`(新增) / `fix`(修bug)
**scope**：用 sop 短名，如 `refactor(tg_sop)`

## 三、避坑清单

| 坑 | 解法 |
|----|------|
| 交叉引用不闭环 | 改完后 grep 检查所有交叉引用是否有效（`FILE:` 路径正确？） |
| 忘记更新版本号 | 改完先扫标题行，搜 `(v\d+)` 或 `(版本)` |
| 编号断链 | 删除整节后强制全文档编号审查（用眼睛扫比搜索可靠） |
| 记忆只改L3不改L2/L1 | 改完后先搜 `global_mem.txt` 中本SOP名，看描述是否仍匹配 |
| 多层级引用导致查找困难 | 最多一级引用（A→B），不搞 A→B→C→D 链 |
| 角色卡与SOP同时修改 | 优先改角色卡（源），再改SOP（引用者），避免反向传播 |

## 四、批量操作模式（v2.0 新增）

> 来源经验：2026-05-05 35/35 SOP文件标准化，6批次完成

### 适用场景
- 多文件需统一加装摘要/门禁/格式
- 模板化操作可复用（同一patch内容应用到多文件）

### 流程
① 全量扫描 → `code_run` grep统计状态（SG=有摘要+门禁, S=有摘要缺门禁, NONE=全缺）
② 分批处理 → 按状态分组，5-10文件一批（防止上下文过长）
③ 模板注入 → 读文件头尾 → `file_patch` 注入摘要+门禁（摘要先，门禁后）
④ 重扫验证 → 每批完成后重扫，确认状态升级（NONE→S→SG）
⑤ 异常修复 → 缺失补丁（如非标准标题修正为 `## 🛑 验证门禁`）
⑥ 门禁自检 → 最终全量扫描，确认目标状态100%达成

### 避坑
| 坑 | 解法 |
|----|------|
| patch失败不知原因 | 每批patch后重扫，确认状态迁移 |
| 门禁标题不统一 | 强制使用 `## 🛑 验证门禁`（无后缀） |
| 非SOP文件被误扫 | 提前确认范围，排除角色卡/subagent/journal |
| 上下文过长 | 每批≤10文件，读头尾不读全文（grep验证即可） |

## 🛑 验证门禁

| 检查项 | 状态 |
|--------|------|
| 全文已读(非局部分析)？ | |
| 版本号已更新？ | |
| 交叉引用(sop名/路径)闭环？ | |
| L2(global_mem)+L1(insight)已同步？ | |
| 编号无断链(删除节后全审查)？ | |
| 角色卡优先改(源)，SOP后改(引用者)？ | |
| 批量操作后全量重扫(100%达标)？ | |

最终裁定：`VERDICT: PASS` / `VERDICT: FAIL`
