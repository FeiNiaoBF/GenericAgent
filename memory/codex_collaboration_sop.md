## 道：三条核心判断
1. **以边界换聚焦** — 明确不做什么/可妥协/不可妥协，结果才聚焦
2. **以信息换准确** — 目标、背景、约束、细节说清，返工越少
3. **以文档和验证换稳定** — 上下文会衰减，共识尽早落文档+测试闭环

## 法：单次任务最小闭环
```
目标、模式、约束、输出、验证
```
推荐骨架：
```text
目标：解决【具体问题】；尽量逼近【最终可用状态】。
模式：【直接推进 / 先诊断 / 先复现 / 先收敛计划】。
约束：这次只解决【范围A】，不扩展到【范围B】；优先【长期可维护/短期可交付】。
输出：先给【判断/方案/计划】，再【落地/继续问关键问题】。
验证：通过【测试/最小验证步骤/验收标准】证明结果有效。
```

## 术：12种高频局面→打法速查

| # | 局面 | 一句话打法 |
|---|------|-----------|
| 1 | 局面未明 | 先诊断不硬改：列假设+缺失信息，按优先级输出问题 |
| 2 | 主线清晰 | 直接推进：读上下文→动手，非关键不问，只拍板级问题集中问 |
| 3 | 长期建设 | 优先长期可维护；短期方案必须标注取舍和后续风险 |
| 4 | 短期止血 | 优先恢复可用；临时方案标注后续治理项 |
| 5 | 不需要兼容 | 直接硬切：不保留兼容层/桥接逻辑，同步调整调用方 |
| 6 | 连续失败 | 先复盘：判断目标错/路径错/执行细节错→给下一轮方案+停掉低收益动作 |
| 7 | 链路越走越深 | 评估停点：已获什么/缺什么/继续投入收益→判断继续/收口/换路/暂停 |
| 8 | 临近收口 | 补验收标准+验证步骤，证明问题确实解决而非碰巧通过 |
| 9 | Bug场景 | 先复现再修复：最小失败测试或复现步骤→确认复现→修→回归验证 |
| 10 | 多轮协作 | 先落文档：目标/约束/共识/待决/下一步结构化记录，持续更新 |
| 11 | 任务复杂 | 先收敛计划：梳理目标/约束/边界/分歧点/路径/步骤再动手 |
| 12 | 发现技术债 | 先挂账不扩散：阻塞→说明原因再处理；不阻塞→记入债务清单(位置/问题/不处理原因/风险/建议时机) |

## 器：四个即用组合

### 方案讨论型
```text
目标：先收敛【方案】，不急着写代码。
模式：必要时先做计划收敛。
约束：提炼真正关键的分歧点，低价值问题不抛。
输出：按"问题、选项、利弊、推荐"格式，我来拍板。
验证：明确解决什么、不解决什么、为什么。
```

### 直接开发型
```text
目标：解决【具体问题】，逼近【最终可用方案】。
模式：任务明确→直接推进。
约束：先读上下文再动手；非关键不问；只解决当前核心问题，不顺手扩展。
输出：先给判断和推荐方案，再落地；必须拍板的按"问题、选项、利弊、推荐"问。
验证：改完后补充测试/最小验证闭环。
```

### Bug修复型
```text
目标：修复【bug】，避免盲改。
模式：一般不需计划收敛（除非牵涉大范围方案调整）。
约束：先复现再修复；不绕开真实问题。
输出：根因、复现方式、修复代码。
验证：先确认稳定复现→修复后通过→补充回归验证。
```

### 独立复查型
```text
目标：独立复查当前结果，不重复主流程。
模式：主流程基本完成后使用。
约束：从架构/接口/前端/文档视角检查稳定性/可维护性/一致性；聚焦复查不重复实现。
输出：风险、遗漏、更优建议 + 已足够不需继续投入的点。
验证：分开列出"必须改"和"可选优化"。
```

## 通用补充句（按需追加）

- `这次只解决当前核心问题，不顺手扩展、不做无关重构。`
- `这次不考虑兼容过渡，直接按目标状态修改；不要额外保留兼容层、桥接逻辑或过渡方案。`
- `如果有需要我确认的问题，请只提最关键的1~3个，并按"问题、选项、利弊、推荐"格式来问。`
- `先列出你的关键假设和缺失信息；能自行拍板的直接继续，不能拍板的最后集中列出。`
- `如果出现兜底方案，不要直接采用。先说明更优正解是什么、为什么暂时不可行；确认后再说风险、局限和回收方式。`

## 五阶段协作流
```
计划(收敛) → 诊断(找问题) → 复现(锁bug) → 执行(直接推) → 复查(独立审)
   ↑              ↑              ↑            ↑            ↑
  模糊任务      局面未明       Bug场景      主线清晰      阶段完成
```
阶段切换触发条件：明显偏航/反复失败/需要重新收敛时回退到计划模式。
